# v3.3 Spec Freeze: Implicit Text

Date: 2026-03-12
Status: frozen-for-implementation
Owner: Emad

## Objective

Add parser-only implicit text behavior in Rust without changing explicit syntax stability.

## Non-Goals

- No serializer rewrite.
- No multiline grammar redesign.
- No change to unknown `word:` handling.
- No change to blank-line block termination behavior.

## Baseline (Current Rust Behavior)

From `intenttext-rust/src/parser.rs`:

- Blank line flushes current block.
- Indented lines (`2+` spaces or tab) are continuation lines for current block.
- `keyword:` lines resolve through keyword registry; unknown keywords still parse as extension blocks with warning diagnostics.
- Unrecognized non-keyword lines become `text` blocks.

## v3.3 Grammar Contract

Line-level parse precedence remains:

1. Code fence handling.
2. Blank-line flush.
3. Continuation line (`2+` spaces or tab).
4. Divider/table/list shorthand handling.
5. `keyword:` handling (including unknown keyword behavior).
6. Implicit text fallback.

Implicit text fallback rule:

- Any non-empty, non-indented line that does not match higher-priority forms MUST produce a `text` block line payload.
- If no current block exists, parser starts a `text` block.
- If current block exists and line is not continuation-eligible, parser flushes and starts a new `text` block.

## Acceptance Examples

### A1: Bare prose line becomes `text`

Input:

```it
Hello world
```

Expected parse shape:

- One block: `type=text`, `content="Hello world"`.

### A2: Mixed explicit and implicit text

Input:

```it
title: Example
This paragraph is implicit text.
```

Expected parse shape:

- `title` block with `content="Example"`.
- `text` block with `content="This paragraph is implicit text."`.

### A3: Continuation remains unchanged

Input:

```it
text: first line
  second line
```

Expected parse shape:

- One `text` block with two lines in content.

### A4: Blank line still terminates block

Input:

```it
text: first

second
```

Expected parse shape:

- First `text` block with `content="first"`.
- Second `text` block with `content="second"` (implicit).

### A5: Unknown `word:` remains unknown keyword behavior

Input:

```it
customword: payload
```

Expected parse shape:

- One block with `type="customword"`.
- Warning diagnostic for unknown keyword.
- No implicit-text reinterpretation.

### A6: Explicit keyword line still wins over implicit text

Input:

```it
section: Scope
```

Expected parse shape:

- One `section` block.

## Edge-Case Matrix

| Case | Input shape                          | Expected behavior                              |
| ---- | ------------------------------------ | ---------------------------------------------- |
| E1   | Empty input                          | No blocks                                      |
| E2   | Whitespace-only line                 | Flush current block, no new block              |
| E3   | Bare prose after explicit block      | New implicit `text` block                      |
| E4   | Bare prose before any explicit block | New implicit `text` block                      |
| E5   | Indented prose with active block     | Continuation line appended                     |
| E6   | Indented prose with no active block  | Ignored as continuation-only line (unchanged)  |
| E7   | Unknown `word:` line                 | Unknown keyword warning + extension-like block |
| E8   | Code-fence body containing `word:`   | Literal code content, not keyword parse        |

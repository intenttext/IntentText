# v3.4 Property Continuation Plan

Target release: `3.4.0`
Status: `proposed`
Owner: `Emad`

## Goal

Improve source readability by allowing property continuation lines for the previous block without changing intent semantics.

Example:

```it
task: Write the architecture document
    | owner: Ahmed
    | due: 2026-04-15
    | note: Must include sequence diagrams and API contracts
```

Equivalent to:

```it
task: Write the architecture document | owner: Ahmed | due: 2026-04-15 | note: Must include sequence diagrams and API contracts
```

## Non-Goals

- No shortcut syntax (`#`, `>`, `-`, `!`, task inline shorthand).
- No multiline content blocks (`end:` redesign).
- No implicit defaults/state blocks (`with:`).
- No change to one-intent-per-line semantics.

## Syntax Contract

A continuation property line is valid only if all rules match:

1. Begins with one or more spaces or tabs.
2. First non-whitespace character is `|`.
3. Remainder matches `key: value` (same property grammar as inline pipe properties).
4. A valid base block exists immediately before it (ignoring no lines).

Invalid continuation lines should produce diagnostics and be treated as plain text or parse warnings according to existing error policy.

## Parser Rules

1. Parse base block line normally.
2. While subsequent lines satisfy continuation contract, append each parsed property to base block.
3. Continuation does not create new blocks.
4. Duplicate property keys keep current merge behavior (last write wins unless keyword-specific override exists).
5. Continuation ends on first non-matching line.

## Edge Cases

- Continuation line at top of file: invalid.
- Continuation following divider/table/code fence/history marker: invalid.
- Continuation after blank line: invalid (no active base block).
- Continuation with empty value (`| owner:`): allowed if inline property parser allows it.
- Mixed tabs/spaces indentation: allowed, but lint warning can be considered later.

## Serializer Policy (v3.4)

- Keep canonical one-line output as default serializer behavior.
- Optional pretty serializer mode can emit continuation form later.
- Do not auto-rewrite existing files in this release.

## Tooling Impact

- VS Code syntax highlighting: treat indented `| key: value` lines as continuation-property tokens.
- Formatter/linter (if present): optional rule to align continuation indentation.
- No runtime API signature changes.

## Test Matrix

- Single block with 1..N continuation lines.
- Mixed explicit inline properties + continuation properties.
- Invalid continuation placements (top/file, after blank line, after divider/history/table/fence).
- Property precedence and duplicate-key behavior.
- Roundtrip parse -> source -> parse stability with canonical serializer.
- Trust/hash smoke test to ensure no unintended semantic changes.

## Rollout Plan

1. Implement in Rust parser.
2. Ensure TS thin layer/Python bridge receive behavior automatically from Rust parse.
3. Add fixtures in core parser tests.
4. Add release note section with before/after readability examples.

## Exit Criteria

- All existing parser/validate/render/trust tests pass.
- New continuation tests pass.
- No regressions in history/trust boundaries.
- No changes required for existing documents.

## Risk

Continuation introduces line-state parsing and potential ambiguity.

## Mitigation

- Narrow grammar (strictly indented + leading `|` + immediate previous block only).
- Keep serializer unchanged by default.
- Ship with explicit diagnostics for ambiguous/invalid continuation usage.
